# Итоговая логика календаря TimeAgree

## Концепция

Система управления временем, основанная на универсальных принципах календарной системы с учетом требований:
- Точное хранение времени в UTC (без привязки к часовым поясам)
- Гибкие правила повторений с исключениями
- Проверка конфликтов при создании
- Целевые события (день рождения, отпуск)
- Напоминания
- Синхронизация между устройствами

---

## 1. Сущность "Событие" (Event)

### Минимальный набор данных:

**Основные поля:**
- `id` - уникальный идентификатор
- `type` - тип события (enum):
  - `ONE_TIME` - одноразовое событие
  - `CYCLIC_WEEKLY` - повторяющееся по дням недели
  - `ALL_DAY` - целевое событие (день рождения, отпуск)
- `title` - название события
- `description` - описание
- `startTime` - UTC время начала (DateTime)
- `endTime` - UTC время окончания (DateTime)
- `timezone` - часовой пояс пользователя (string, например "Europe/Moscow")
- `startAt` - дата создания события (DateTime)
- `updatedAt` - дата последнего обновления (DateTime)

**Статус (enum):**
- `CONFIRMED` - подтверждено
- `CANCELLED` - отменено
- `DRAFT` - черновик
- `ARCHIVED` - архивировано (после завершения)

**Целевые события:**
- День рождения (весь день занят, независимо от часового пояса)
- Отпуск (с даты по даты, весь день занят)

**Отличие от обычных событий:**
- Для `ALL_DAY`:
  - `startAt` = дата начала отпуска
  - `endTime` = дата окончания отпуска
  - `startTimeLocal` и `endTimeLocal` = NULL (не используются)
  - `dayOfWeek` = NULL
  - При проверке занятости берётся весь день независимо от часовых поясов

## 2. Логика времени

### 2.1 Единый формат хранения

Все время в базе данных хранится в UTC (DateTime в UTC):
```typescript
{
  startTime: '2026-03-05T07:00:00Z',  // 10:00 по Москве
  endTime: '2026-03-05T16:00:00Z'   // 19:00 по Москве
  timezone: 'Europe/Moscow'
}
```

### 2.2 Отображение с учетом часового пояса

Когда пользователь открывает календарь:
1. Получает UTC время из базы данных
2. Определяет свой часовой пояс (из профиля или автоматически)
3. "Натягивает" UTC время на часовой пояс:
   ```typescript
   const localStart = new Date(startTime.getTime() + timezoneOffset * 60000)
   const localEnd = new Date(endTime.getTime() + timezoneOffset * 60000)
   ```
4. Отображает локальное время в интерфейсе

**Пример:**
- UTC: 07:00 → Москва (UTC+3): 10:00 ✓
- UTC: 16:00 → Лондон (UTC+0): 16:00 ✓
- База данных оста неизменной: 07:00 UTC

### 2.3 Целевые события (ALL_DAY)

Для целевых событий (день рождения, отпуск):
- `startAt` и `endTime` - это ДАТА начала и конца отпуска
- При проверке занятости используется весь день независимо от времени начала и конца
- `startTimeLocal` и `endTimeLocal` не используются

## 3. Логика повторений

### 3.1 Шаблон правила

**Хранится в базе:**
```typescript
{
  type: 'CYCLIC_WEEKLY',
  dayOfWeek: 4,  // Четверг (JavaScript: 4, в отображении это пятница)
  startTimeLocal: '09:00',
  endTimeLocal: '18:00',
  description: 'Работа',
  // startAt, endTime - NULL
}
```

**Генерация на месяц:**
- При просмотре месяца система берёт шаблон
- Вычисляет все даты месяца, которые соответствуют правилу
- Временно "разворачивает" - показывает как обычные слоты для выбранного месяца
- Не сохраняет разрозненные даты в базу (показывает из шаблона)

### 3.2 Разрывы в серии

**Пример:**
Пользователь создал правило "Работа каждый Пн 9-18", но в следующий понедельник встреча на час позже.

**Решение:**
1. Создает копию конкретной встречи на изменённое время
2. Помечает эту конкретную дату в исходном правиле как "изменённую" (`isModified: true`)
3. При генерации на следующий месяц:
   - Для обычных дат - использовать оригинальное правило
   - Для "изменённых" - использовать изменённое правило
4. Остальные встречи в серии остаются по старому правилу

**Структура:**
```typescript
{
  type: 'CYCLIC_WEEKLY',
  dayOfWeek: 4,
  startTimeLocal: '09:00',
  endTimeLocal: '18:00',
  description: 'Работа',
  overrides: [
    {
      date: '2026-03-10',  // следующий понедельник
      startTimeLocal: '10:00',  // на час позже
      endTimeLocal: '19:00',
      reason: 'Перенес с 9:00-18:00'
    }
  ]
}
```

## 4. Проверка конфликтов

### 4.1 Правило пересечения

**Два события конфликтуют, если:**
- `startAt2 < endAt1` (второе начинается раньше окончания первого)
- `endAt2 > startAt1` (второе заканчивается позже начала первого)

**Реакция (режимы):**
- **Жесткий режим**: запрещает создание события с ошибкой "В это время вы уже заняты"
- **Мягкий режим**: разрешает создание, но отмечает как конфликт (цвет/иконка)
- **По умолчанию**: мягкий режим

### 4.2 Конфликт типов событий

**Приоритет конфликтов:**
1. `ALL_DAY` > `CYCLIC_WEEKLY` > `ONE_TIME`
2. Утверждённые события > черновики
3. Конфликты > без конфликтов

**Пример:**
- Целевое событие (отпуск) имеет приоритет над обычными встречами

## 5. Логика отображения (Визуализация)

### 5.1 Сетка времени

**Режимы отображения:**
- Месяц: дни × часы
- Неделя: часы × дни

**Размеры блоков:**
- 1 час = 50 пикселей высоты
- 30 минут = 25 пикселей
- 15 минут = 12.5 пикселей

### 5.2 Позиционирование

**Календарь (месяц):**
- Каждый день разделен на ячейки по часам
- Блок события занимает одну или несколько ячеек пропорционально длительности
- Начало блока = час:00 + (минуты / 60) пикселей от верха ячейки

**Неделя (часы × дни):**
- Горизонтальная таблица: дни недели (оси X) × часы (оси Y)
- Каждая строка = один час
- Слоты занимают несколько строк

### 5.3 Наложение событий

**Правило:**
- Если в один час попадают два события:
  - Делит ширину ячейки пополам
  - Размещает события рядом (слева/справа)
  - Не накладывает друг на друга

**Конфликты:**
- Красный цвет с иконкой конфликта
- Оранжевый цвет с мягким конфликтом

### 5. Границы периодов

**При просмотре месяца:**
- Если событие не помещается в ячейку дня, обрезается
- Добавляется подпись "+ еще X"
- X = количество слотов, не поместившихся

## 6. Логика уведомлений

### 6.1 Структура напоминания

```typescript
{
  slotId: string,
  remindAt: DateTime,  // когда отправить
  remindVia: 'push' | 'email' | 'both',
  message: string,
  status: 'pending' | 'sent' | 'failed',
  sentAt?: DateTime
}
```

### 6.2 Внутренняя очередь задач

**Фоновый процесс:**
- Проверяет текущее время каждую минуту
- Сверяет с очередью задач
- При совпадении времени - отправляет уведомление
- Отмечает как `sent` чтобы не дублировать

### 6.3 Создание напоминания

**При создании слота с напоминанием:**
- Добавляется запись в `/api/reminders`
- При сохранении слота создается связанное напоминание
- При удалении слота - удаляется связанное напоминание

---

## 7. Синхронизация устройств

### 7.1 Источник истины
- Все данные хранятся на сервере (PostgreSQL)
- Устройки (选择的主题, язык, часовой пояс) - только отображают копию

### 7.2 Синхронизация изменений
- Телефон отправляет новые данные на сервер при каждом изменении
- Сервер сообщает всем другим устройствам пользователя
- Устройки автоматически синхронизируются

### 7.3 Конфликт синхронизации
- Если пользователь изменил одно и то же событие одновременно на двух устройствах:
  - Принимается "последняя запись побеждает"
  - Или предлагается выбрать версию вручную

## 8. Итоговая модель базы данных

### 8.1 Таблица `Slot`

```typescript
{
  id: string
  userId: string
  groupId: string
  
  // Тип события
  type: 'ONE_TIME' | 'CYCLIC_WEEKLY' | 'ALL_DAY'
  title?: string
  description?: string
  
  // Время (всегда в UTC)
  startTime: DateTime      // для ONE_TIME и ALL_DAY
  endTime: DateTime        // для ONE_TIME и ALL_DAY
  
  // Параметры повторения (только для CYCLIC_WEEKLY)
  dayOfWeek?: number          // 0=Вс, 1=Пн, 2=Вт, 3=Ср, 4=Чт, 5=Пт, 6=Сб
  startTimeLocal?: string      // формат "HH:MM"
  endTimeLocal?: string        // формат "HH:MM"
  overrides?: Override[]      // исключения для отдельных дат
  
  // Статус
  status: 'DRAFT' | 'CONFIRMED' | 'CANCELLED' | 'ARCHIVED'
  
  // Временные метаданные
  startAt: DateTime
  createdAt: DateTime
  updatedAt: DateTime
  
  // Напоминания
  reminders: Reminder[]   // связь с таблицей Reminder
  
  // Конфликты
  conflicts: Conflict[]    // список конфликтов с другими событиями
  
  // Для ALL_DAY (целевые события)
  isAllDay: boolean      // флаг целевого события
}
```

### 8.2 Таблица `Reminder`

```typescript
{
  id: string
  slotId: string  // связь с таблицей Slot
  remindAt: DateTime  // когда отправить
  remindVia: 'push' | 'email' | 'both'
  message: string
  status: 'pending' | 'sent' | 'failed'
  sentAt?: DateTime
  createdAt: DateTime
}
```

### 8.3 Таблица `Conflict`

```typescript
{
  id: string
  slotId: string
  conflictingWithId: string  // ID конфликтующего слота

```

## 9. Алгоритм работы

### 9.1 Создание события

```typescript
// 1. Валидация обязательных полей
if (!title || !type || !startTime || !endTime) {
  throw new Error('Missing required fields')
}

// 2. Проверка на конфликты
const conflicts = await checkConflicts({
  startTime,
  endTime,
  userId,
  groupId,
  slotType: type,
  excludeSlotId: undefined
})

if (conflicts.length > 0 && conflictMode === 'hard') {
  throw new Error('В это время вы уже заняты')
}

// 3. Создание события
const slot = await createSlot({
  userId,
  groupId,
  type,
  title,
  description,
  startTime,
  endTime,
  dayOfWeek,
  startTimeLocal,
  endTimeLocal,
  overrides
})

// 4. Если есть напоминание, создать запись
if (remindAt) {
  await createReminder(slot.id, remindAt, remindVia, message)
}

// 5. Создание конфликтов
for (const conflict of conflicts) {
  await createConflict(conflict, slot.id)
}
```

### 9.2 Генерация на месяц

```typescript
// Для отображения месяца в календаре
async function generateMonthSlots(year: number, month: number): Slot[] {
  const slots: Slot[] = []
  
  // Получаем шаблоны правил повторения из базы
  const rules = await getCyclicRules(groupId, userId)
  
  const firstDay = new Date(year, month, 1)
  const lastDay = new Date(year, month + 1, 0)
  
  // Для каждого шаблона правила:
  for (const rule of rules) {
    // Для CYCLIC_WEEKLY:
    if (rule.type === 'CYCLIC_WEEKLY') {
      // Для каждого дня месяца:
      for (let date = firstDay; date <= lastDay; date.setDate(date.getDate() + 1)) {
        // Если день совпадает с правилом:
        if (date.getDay() === rule.dayOfWeek) {
          // Генерируем разрозненную запись на эту дату
          slots.push({
            id: genUUID(),
            userId: rule.userId,
            groupId: rule.groupId,
            type: 'ONE_TIME',
            title: rule.title,
            description: rule.description,
            startAt: new Date(`${date.toISOString().split('T')[0]}T${rule.startTimeLocal}:00`),
            endAt: new Date(`${date.toISOString().split('T')[0]}T${rule.endTimeLocal}:00`),
            dayOfWeek: undefined,  // разрозованная запись без dayOfWeek
            startTimeLocal: rule.startTimeLocal,
            endTimeLocal: rule.endTimeLocal,
            isGenerated: true  // флаг, что это сгенерированная запись
            overrides: rule.overrides
          })
        }
      }
    }
  }
  
  return slots
}
```

### 9.3 Проверка конфликтов

```typescript
async function checkConflicts(startTime, endTime, userId, groupId, excludeSlotId?: string): Conflict[] {
  // Получаем все слоты пользователя в группе
  const userSlots = await getUserSlots(userId, groupId)
  const conflicts: Conflict[] = []
  
  for (const slot of userSlots) {
    // Игнорируем удаляемые или отменённые слоты
    if (slot.status !== 'CONFIRMED') continue
    if (excludeSlotId && slot.id === excludeSlotId) continue
    
    // Проверка пересечения
    if (startTime < slot.endTime && endTime > slot.startAt) {
      // Конфликт!
      conflicts.push({
        id: genUUID(),
        slotId: slot.id,
        conflictingWithId: slot.id,
        type: 'time_conflict',
        resolved: false,
        createdAt: new Date()
      })
    }
  }
  
  return conflicts
}
```

## 10. Приоритеты конфликтов

1. **ALL_DAY > CYCLIC_WEEKLY > ONE_TIME** - целевые события имеют приоритет
2. **CONFIRMED > DRAFT > CANCELLED > ARCHIVED**
3. **Жесткий режим**: запрет создания при конфликтах
4. **Мягкий режим**: пометка конфликтов вместо запрета
5. **По умолчанию**: мягкий режим

## 12. Резюме

Система управления временем TimeAgree строится на фундаментальных принципах:

1. **Универсальное время** - хранение в UTC для избежания проблем с часовыми поясами
2. **Гибкие правила** - хранение только шаблонов, генерация на лету
3. **Исключения** - возможность изменить отдельные встречи без изменения всего правила
4. **Проверка конфликтов** - защита от пересечений (hard/soft режимы)
5. **Целевые события** - день рождения, отпуск (занимают весь день)
6. **Визуализация** - правильное отображение с учетом часовых поясов, наложение блоков
7. **Напоминания** - автоматические уведомления о предстоящих событиях
8. **Синхронизация** - все данные на сервере, синхронизация устройств
